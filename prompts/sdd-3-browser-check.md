仕様書駆動開発のフェーズ3として、実装結果レポートをもとにブラウザでの動作確認を行い、問題があれば修正を行いなさい。

## ルール
- あなたがClaudeCode、Codexではない場合は処理を中止し、「この処理はClaudeCode、Codexでのみ実行可能です。」とユーザーに報告しなさい。
- ブラウザ操作についての知識はdocs/browser-knowledge.mdを参照しなさい。作成されていない場合は作成しなさい。
- 開発サーバーが起動しているポートを勝手に変えない
- 各テンプレートやドキュメントが取得できない場合は直ちにユーザーに報告し、作業を中止しなさい。
- 回復不可能な問題が発生した場合は処理を中止し、ユーザーに報告しなさい。
- 発生している問題を解決するためにignoreをつけたり、手順をskipすることは許されない。
- 問題が解決したら、ユーザーに報告しなさい。
- 問題が解決しない場合は、ユーザーに報告しなさい。
- チェックを通すための暫定的な処理をソースコードに追加してはいけない。問題は本質的に解決されなければならない。
- もし大きな修正が必要な場合(引数やテーブル設計の変更など)は、ユーザーに確認を求めなさい。
- ログインが出来ない場合はユーザーに報告しなさい。
- 可能な限りデバッグログイン(プロジェクトにより特定のURLにアクセスすれば一発でログイン出来るデバッグ用URLが実装されていることがある)を使用しなさい。
- アーキテクチャを理解した上で修正を行うこと。
- アーキテクチャは(https://raw.githubusercontent.com/noaqh-corp/dev/refs/heads/main/docs/architecture.md)から取得しなさい。
- スクリーンショットは`spec/{feature_number}-{feature_name}/screenshots/{screenshot_name}.png`に保存しなさい。
- ログがlogs/に出力されている場合があります。調査に活用しなさい。
- 問題を解決するための開発サーバーの再起動は最後の手段にしなさい。基本的には時間の無駄です。
- スクリーンショットを作成し、結果レポートを更新しながら作業をすすめる。
- 基本的にはシードデータからログイン方法を見つけるべき。新規登録は新規登録が目的の動作確認以外では不要であるべき。


## 手順
1. すでに開発サーバーが起動しているか確認する。起動していない場合のみ開発サーバーを起動しなさい。起動する場合はシンプルに`bun run dev`を実行しなさい。
2. `spec/{feature_number}-{feature_name}/`を読み込み、実装結果レポートの内容を完全に理解しなさい。ブラウザ確認結果レポートテンプレートを参考に、ブラウザ確認結果レポートを空で作成。
3. 実装結果レポートに記載されている動作確認の手順を実行しなさい。
4. 実装結果レポートに記載されている動作確認の結果が正常かを確認しなさい。
5. 問題があれば修正しなさい。
6. 修正が完了し、関連機能が動作することを確認したらlintを実行し、問題があれば修正しなさい。
7. 問題が解決したら、ブラウザ操作についての知識について追加、編集するべき箇所があればシンプルな箇条書きでdocs/browser-knowledge.mdに追加、編集しなさい。ログイン方法やアカウント情報、シードデータの作成方法、データ初期化方法はテンプレートに従いなさい。

8. 問題が解決したら、ブラウザ確認結果レポートをブラウザ確認結果レポートテンプレートに従い漏れがないか確認し、ユーザーに報告しなさい。
