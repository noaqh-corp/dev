私は最小の手間で必要十分かつ正確で最低限の情報を提供するgitコミットメッセージを作成したい。
手順に従い、gitコミットメッセージを作成しなさい。

## 手順
1. コミット対象の絞り込み:
1-a. これまでの会話履歴がある場合:会話履歴を元にして何を対応したか理解し、どのファイルのどの部分が対象かを理解する。これまでの会話履歴がない場合は、対象となるファイルを指定する。
1-b. これまでの会話履歴がない場合かつ、ステージングエリアにのみ変更がある場合:ステージングエリアにある変更を対象とするが、ステージングエリアに漏れがないか確認する。
1-c. これまでの会話履歴がない場合かつ、ステージングエリアにのみ変更とステージングされてない変更がある場合:ステージングエリアにある変更とステージングされてない変更を対象とするが、ステージングエリアに漏れがないか確認する。
2. コミットメッセージの作成を行い、ユーザー確認テンプレートに従いユーザーに確認を行う。
3. ユーザーがコミットメッセージを承認した場合:
3-a. コミットメッセージをコミットテンプレートに従い再作成し、ユーザーに確認を行う。
3-b. コミットを行う。

## ユーザー確認テンプレート

```
コミット準備が完了しました。コミットメッセージを確認してください。承認する場合は「y」、修正する場合はどのように修正するかを入力してください。

---
{コミットメッセージ}
---

対象ファイル: 
    - {対象ファイル1}
    - {対象ファイル2}
    - ...
    - {対象ファイルn}


```


## コミットテンプレート

```
{emoji} {100文字以下で簡単に内容がわかるようになる短いサマリー。体言止めで冗長な表現は避け、句読点は使わず、簡潔に記載しなさい。}
{必ず空行}
{変更内容の詳細な説明 なぜ変更したか、詳細な背景、影響範囲}


```

## 良い例

```
🐛 %割引クーポンの割引対象額が送料込み価格ではなく商品価格のみになっていた問題を修正

変更前の状態:
    - クーポンは送料込み価格で割引対象額を計算していた
変更後の状態:
    - クーポンは商品価格で割引対象額を計算していた

```

## 悪い例1

```
🐛 修正 // 短すぎて意味がわからない

変更前の状態:
    - クーポンの割引価格に問題があった // 変更前の状態がわからない
変更後の状態:
    - クーポンの割引価格に問題を修正した // 変更後の状態がわからない
```

## 悪い例2

```
🐛 クーポンには割引対象の価格を計算する機能があり、これは送料込み価格ではなく商品価格のみになっていた問題を修正した // 冗長な表現が多すぎる。%割引クーポンのみ対応したという重要な情報が喪失。クーポンには割引価格を計算する機能があるのは自明であり、これを記載する必要がない。

変更前の状態:
    - クーポンの割引価格に問題があった // 変更前の状態がわからない
変更後の状態:
    - クーポンの割引価格に問題を修正した // 変更後の状態がわからない
```
