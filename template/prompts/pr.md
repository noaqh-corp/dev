現在のブランチで作成した変更を手順に従い、PRとして作成しなさい。もし、現在のブランチがfeature/*ブランチであれば、feature/*ブランチからmainブランチにPRを作成しなさい。もし、現在のブランチがmainブランチであればメインブランチではPRは作成できないというエラーをユーザーに知らせて、作業は中止しなさい。

## ルール
- 必要な情報が不足していればユーザーに要求する
- コミット漏れと思われるファイルがあればユーザーにコミット漏れの存在を知らせて、作業は中止しなさい。
- 対応する実装計画書が存在する場合はPRには実装計画書の内容を貼り付ける。実装計画書のコミットは禁止。
- 対応する実装レポートが存在する場合はPRには実装レポートの内容を貼り付ける。実装レポートのコミットは禁止。

## 手順
1. 現在のブランチをチェックし、feature/*ブランチ、もしくはbugfix/*ブランチでない場合は「現在のブランチがfeature/*ブランチかbugfix/*ブランチではないため、PRを作成できません。」というエラーをユーザーに知らせて、作業は中止する。
2. コミット漏れと思われるファイルがあればユーザーにコミット漏れの存在を知らせて、作業は中止しなさい。
3. ユーザー確認テンプレートをもとに、ユーザーに確認を行う。対応する実装計画書や実装レポートが絞りきれない場合は、ユーザーに対応する実装計画書や実装レポートがどれかを質問する。自分で作ったり勝手にコミットはしない。
4. ユーザーが確認を承認した場合:
4-a. PRを作成する。
4-b. PRを作成したらユーザーにPRのURLを知らせる。

## PR名について
必ずNOA-[数字]-[PRの内容]の形式でPR名を作成する。数字はlinearのISSUE番号になる。したがって、LinearのISSUE番号が分からない場合はユーザーに質問する。

## PRテンプレート

```

## 背景とリンク
{
この変更が必要になった背景や変更の目的を書いてください。
linear のリンク、Slack threads など関連するリンクを貼ってください。
}

## Impact on end user

- [ ] ユーザーへの影響あり
- [ ] ユーザーへの影響なし

{
必ずどちらかにチェックを入れてください。

}

## 変更内容
{
具体的な変更内容について詳しく説明してください。スクリーンショットがある場合は貼ってください。

ただし、上記ラベルを使用するときはレビュアーと相談し、合意を取ってください。PR上にも意思決定の経緯などがわかるものを残してください。
}

## 変更内容の除外
{ なんらかの理由で今回の変更のスコープから外すものがある場合は理由と対象を説明してください }

## 依存するPR
{
このPRに依存している、もしくは依存されているプルリクエストを列挙してください。

CLI、Front、およびAPIサーバーは非同期にデプロイされることに注意してください。
常にサービスが正しい動作をすることを保証するため、どのプルリクエストが先にマージされなければいけないか、
順番を気にする必要がある場合があります。

1. 下位互換性のある方法でAPIを変更した場合、依存関係をあまり気にする必要はありません。
   しかし、例えばAPIがデプロイされる前に、ユーザーが更新されたページを開いてしまう可能性があります。
   これが問題になる可能性がある場合は、PRを分割して、先にAPIの変更をマージしてください。
2. APIと他のコンポーネントに後方互換性のない破壊的変更を加えたい場合は、その順序をよく考えてください。
   例えば、ウェブとCLIを新旧のAPIで動作するように変更し、APIの変更点をマージし、クリーンアップとして
   ウェブとCLIから後方互換性のあるコードを削除するという方法があります。
}
```
